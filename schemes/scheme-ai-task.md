# Схема — ai-task.md

## ІНСТРУКЦІЯ ДЛЯ AI

`ai-task.md` — файл задач проєкту. Може містити кілька задач одночасно.
Кожна задача має статус: В роботі | Закрита

---

## LIFECYCLE ЗАДАЧІ

```
Користувач описує задачу
        ↓
AI створює запис зі статусом «В роботі»
        ↓
AI додає нагадувач в кінець КОЖНОЇ відповіді
        ↓
Користувач каже "готово" або "є правки: [опис]"
        ↓
AI перевіряє рівень змін (критичні / м'які)
        ↓
[Якщо критичні] → просить підтвердити оновлення документації
[Якщо м'які]    → одразу закриває
        ↓
Статус → Закрита
```

---

## Нова тема без явного «готово»

Якщо користувач **не** сказав «готово» / «є правки» і **перемикає контекст** (нова тема, інший запит, ніби вже інша задача):

1. **Не** закривай попередню задачу мовчки і **не** ігноруй її на користь нової.
2. **Попередь** про незакриті задачі (назви, статус, критичність).
3. **Запитай явно**: відкрити нову задачу поверх поточних чи спочатку закрити/уточнити стару — див. також розділ «Відкриття нової задачі» нижче.
4. Продовжуй (новий запис у `ai-task.md` або робота далі) **лише після** відповіді користувача.

---

## РІВНІ ЗМІН

### Критичні — не закривати без підтвердження документації
- Зміни в БД (нові таблиці, нові поля, видалення полів, ALTER)
- Новий модуль в cactus/
- Зміни в Warning Zone файлах
- Нові бібліотеки в system/library/
- Зміни архітектури або роутингу
- Нові залежності (composer, npm)

### М'які — закривати одразу після "готово"
- CSS / JS правки
- Правки шаблонів (`.twig` / `.tpl`)
- Текстові зміни
- Дрібні багфікси без змін архітектури

---

## НАГАДУВАЧ (додавати в кінець КОЖНОЇ відповіді поки є відкриті задачі)

### Якщо одна відкрита задача:
```
---
Задача відкрита: [назва]
[КРИТИЧНО: зміни в БД / новий модуль / ядро — документацію оновлено?]
Скажи "готово" або "є правки: [опис]" щоб закрити.
```

### Якщо кілька відкритих:
```
---
Відкриті задачі (N):
[КРИТИЧНО] [назва] — документація не підтверджена
[М'ЯКО] [назва] — м'які зміни
[В РОБОТІ] [назва] — поточна
Скажи "готово [назва]" або закрий всі: "готово всі"
```

---

## ВІДКРИТТЯ НОВОЇ ЗАДАЧІ (якщо є незакриті)

Стосується і явного запиту нової задачі, і **перемикання теми без «готово»** (див. розділ вище).

Якщо користувач описує нову задачу, а є незакриті — **спочатку** повідом і **запитай**:

```
УВАГА: є незакриті задачі:
- [назва 1] — критичні зміни
- [назва 2] — в роботі

Відкрити нову задачу поверх них? (так / спочатку закрити старі)
```

Якщо користувач каже "так" — відкриваєш нову не закриваючи старі.
Не закривай старі задачі автоматично — тільки користувач вирішує.

---

## НОВА СЕСІЯ (якщо є незакриті задачі)

На початку сесії якщо є незакриті задачі — одразу повідом:

```
УВАГА: незакриті задачі з попередньої сесії:
- [назва 1] — критичні зміни, документація не підтверджена
- [назва 2] — в роботі

Продовжуємо роботу чи спочатку закриємо?
```

## ШАБЛОН ЗАПИСУ ЗАДАЧІ

```markdown
---

## [назва задачі]
> Статус: В роботі
> Відкрита: YYYY-MM-DD
> Закрита: —
> Рівень змін: Критичні / М'які

### Що зробити
[Опис — 1-3 речення]

### Scope — що чіпаємо
| Файл | Що змінюємо |
|------|-------------|
| [шлях] | [опис] |

### Scope — що НЕ чіпаємо
- [файл] — [причина]

### Файли та функції з якими працювали (оновлювати під час задачі)

Фіксуй **кожен** файл, до якого торкався код або який створив AI. Оновлюй таблицю по ходу, не лише при закритті.

| Шлях | Створено (так/ні) | Функції / методи / область змін | Нотатка |
|------|-------------------|----------------------------------|---------|
| `path/to/file.php` | ні | `methodName()`, блок налаштувань | змінено логіку X |
| `catalog/model/cactus/foo.php` | так | увесь файл / ключові методи | новий файл у задачі |

**Правило для DRY:** якщо плануєш винести спільний код — спершу звірся з цією таблицею та з Scope задачі; рефакторинг під DRY без розширення scope — переважно по рядках з **Створено: так** або явно зазначених у Scope.

### Warning Zone на шляху
- [файл] — [нагадування] або **Немає**

### Definition of Done
- [ ] [критерій 1]
- [ ] [критерій 2]

### Лог змін
#### [дата час] — [короткий опис]
- [змінено/створено/видалено]: [шлях]

### Правки користувача
[Те що користувач зробив сам поза AI — вписати перед закриттям]

### Результат
[Заповнюється при закритті]
```

---

## ЗАКРИТТЯ ЗАДАЧІ

При закритті AI:
1. Перевіряє, що секція **«Файли та функції з якими працювали»** відображає фактичні зміни (у т.ч. усі **створені** файли з позначкою **так**).
2. Заповнює "Результат"
3. Міняє статус на Закрита
4. Ставить дату закриття
5. Якщо критичні зміни — фіксує що документацію підтверджено
6. Викликає /update_ai_doc
